3. tenantName / companyName / botName fallback 逻辑
────────────────────────────────────────────────────────────────

V1 没加 tenant.companyName / tenant.botName 字段 (避免 DB migration)
当前 fallback:

  resolveStarterTemplate(template, ctx) {
    companyName: ctx.companyName ?? ctx.tenantName ?? '本公司'
    tenantName:  ctx.tenantName  ?? ctx.companyName ?? '本公司'
    botName:     ctx.botName     ?? 'AI 智能客服'
  }

  seedCommonFaqs 调用时 ctx:
    {
      tenantName: tenant.name,        // 从 tenant 表
      companyName: tenant.name,        // 同上 (V1 没单独字段)
      botName: 'AI 智能客服',           // V1 默认值
    }

实测验证 (Beauty Demo tenant 99):
  faq.answer = "我是 {{companyName}} 的 {{botName}}"
  resolveStarterTemplate 替换后:
  → "我是 Beauty Demo Tenant 的 AI 智能客服"

V2 建议:
  - 加 tenant.bot_name varchar (default NULL)
  - 加 tenant.company_name varchar (default NULL)
  - 后台让租户自定义 bot 名 (e.g. "美美" / "小爱")

9. 风险点
────────────────────────────────────────────────────────────────

R1 (中) · tenant 5 历史 customized starter 数据未清理
  老 FAQ 仍含 FAhubX 等硬编码 (不是代码 bug · 是 SQL 数据)
  影响: tenant 5 客户问 "你是谁" 仍可能答 "我是专为 FAhubX 服务的小助手"
  建议: 跑 reseed-tenant-starter.mjs 5 重新灌 (覆盖) 或 admin 后台重新 AI 优化

R2 (低) · resolveStarterTemplate 不处理转义
  e.g. tenant.name = "Acme {{evil}} Co." → 占位符替换不会再次解析
  当前实施安全 (只 replace {{companyName}}/{{tenantName}}/{{botName}} 三个)
  但租户名含 special chars 时显示可能怪
  建议 V2: 加白名单字符过滤 (替换前 sanitize)

R3 (低) · variants tags 数组长度
  16 条 starter FAQ × 平均 4 variants = 64 个 var:* tags
  Postgres text[] 单元素长度无 hard limit · 64 个元素也 OK
  但每条 FAQ tags 数 = 3 (基础) + ~4 (variants) ≈ 7 · 仍在合理范围
  实测: tenant 99 重灌 55 条 starter · DB 写入 < 100ms

R4 (低) · sanity check 用 KB 名子串匹配
  productNamesInTenant 用 includes 检测 LLM 输出
  例: KB 名 "AI" (2 字) 可能误命中 LLM 输出 "AI 智能客服"
  当前 length >= 3 守卫部分缓解 · 但 KB 名 "AI Bot" (>=3) 仍可能误伤
  建议 V2: 加分词匹配 · 跳常用词

R5 (无) · 跨 tenant 数据泄漏
  occurence: starter customize 仅 read 当前 tenant 自己的 KB
  代码 SQL: WHERE tenantId=$tenantId 强约束

## 8. 关键决策档案 (**别重新讨论, 照做**)

| 决策 | 做法 | 原因 |
|---|---|---|
| 定时器 | 所有都 `setInterval`, 不用 BullMQ repeatable | M3 已这样, 保持一致 · BullMQ 装了但没用 · 用户接受偏离 |
| AI key 存储 | DB `ai_provider.api_key_encrypted` 密文 | 多租户 dev 方便 · 生产 `data/config/ai-providers.json.backup` 只做人工导出 |
| 主密钥来源 | `APP_ENCRYPTION_KEY` env (32B hex) | M6 用 `EnvMasterKeyProvider` · M10 计划换 `MachineBoundMasterKeyProvider` 派生自机器指纹 |
| AI 模式 | 默认关 · `ai.text_enabled=false` | §B.4 降级矩阵: AI 关时 content_pool 原文 |
| Pair null-proxy | 两 null 视为同组 · 拒配对 | §B.15 保守规则 · dev 限制明告 |
| 养号日历 tick | 1h setInterval · 跨天推 day | §B.2 时刻表 + §B.20 status phase gate 严守 |
| Status phase gate | Phase 0-1 禁 · Phase 2 每 3 天 ≤1 · Phase 3+ 每天 ≤1 | §B.20 (§B.2 Day 4 原文"发第一条 Status" 已去掉) |
| 健康分 window | 默认 30 天滚动 | 防历史永久扣分 · `health.scoring_window_days` 可配 |
| Dry-run 默认 | `health.dry_run=false` (生产真降级生效) | 首次 rollout 必手动开 72h |
| 告警 channel | 只桌面 (node-notifier) | email / Telegram 留 interface V1.1 加 |
| 日志脱敏 | `formatters.log` hook 递归, 不用 pino `redact.paths` | pino `*.apiKey` 只 1 层 · 深度嵌套失效 |

## 11. 常见陷阱 · 别踩

1. **worktree 的 cd 不粘**. 所有多步骤命令用 `cd "/c/AI_WORKSPACE/Whatsapp Auto Bot/..." && <cmd>` 单行.
2. **`netstat //IM node.exe`** 会杀 FAhubX. 只用 PID.
3. **TypeORM JSONB 字段传 null** → `_QueryDeepPartialEntity` 类型抱怨. 用 `any` 绕开有注释说明原因即可.
4. **Jest + Baileys ESM** → 每个测试文件顶部 `jest.mock('../baileys/baileys.service', () => ({ BaileysService: class {} }))` 否则 parse 不动.
5. **Nest route 顺序**: `:accountId` 会匹配字面字符串如 `settings`. 静态 path 的 `@Get/@Post` 必须在 `:param` 之前声明.
6. **pino `formatters.log` hook** 深拷贝会掉性能. 测过 OK 但 max depth 设了 8 层. 深嵌套日志会截断 — 大多数情况 OK.
7. **TypeORM QueryBuilder `orIgnore`** 返 `identifiers=[]` 表示冲突. 读 `result.identifiers.length > 0` 判断真插入.
8. **Shell 里 Python `python -c` stdout 管道有时挂** (Windows + bash). Token 失效后要重 login, 不然 `json.load` 报 KeyError.
9. **packages/backend/data/** 不要 add, 是运行时产物. `.gitignore` 已豁免 `data/assets/_builtin/` 但其他 `data/*` 禁.
10. **AES-256-GCM 密文格式** `gcm:v1:{iv}:{ct}:{tag}` 严格 5 段. 第 2 段 `v1` 是版本号, 将来轮换算法换 `v2`.
